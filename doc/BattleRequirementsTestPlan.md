# Battle Requirements Test Plan

**Version:** 1.0 (2026-07-15)
**Companion to:** [BattleRequirements.md](BattleRequirements.md) (v1.0 baseline)

A requirements traceability matrix: one row per requirement, mapping it to the tests that exist
today, the verification method that fits it, and the scope of work to close the gap.

## How this was built

Every requirement was mapped against the actual test suites (`E2E.Tests`, `Coop.Tests`,
`GameInterface.Tests`, `Coop.IntegrationTests`) **and** the implementation. Every "covered" or
"partial" claim was then adversarially re-checked (would the cited test *fail* if the requirement
were violated, or does it only exercise adjacent machinery?), and every "no coverage" claim on
implemented behavior was countered by an independent search for missed tests. Several claims were
downgraded by that pass — notably the victory/defeat tests (BR-070/071), which inject the battle
outcome rather than derive it, and the surrender/retreat citations (BR-052/053/060), which test
adjacent machinery.

## How to read the matrix

| Column | Meaning |
|---|---|
| **Impl** | Is the required behavior in the code today? `Yes` / `Partial` / `No` (requirement is target-state; tests are blocked on implementation). |
| **Coverage** | Do existing tests assert this requirement? `Covered` / `Partial` / `None` — judged against the requirement text, post-verification. |
| **Method** | Cheapest verification method that genuinely verifies it: `unit` (Coop.Tests/GameInterface.Tests, xUnit+Moq) · `e2e` (E2E.Tests headless two-instance harness) · `integration` · `manual` (live 2-client) · `docker` · `design-review` · `deferred`. |
| **Effort** | Scope to close the gap. `S` ≤ ~1 day on an existing harness pattern · `M` 1–3 days, needs new harness capability or multi-component setup · `L` needs the feature implemented first, live multi-client validation, or new test infrastructure. |

**Harness limits that shape the estimates:** the E2E mock mesh routes messages (`IMessage`) but
not packets (`IPacket`), so packet-level behavior (agent movement) is not E2E-testable today; and
engine-level outcomes inside a live `Mission` (actual spawning, visibility, AI/formation release)
can only be confirmed with live clients.

## Headline numbers

- **59 requirements** → **5 covered · 34 partial · 16 none · 4 deferred** (Section 5, post-MVP).
- **Gap-closing scope: 25 S · 18 M · 7 L** (50 requirements need test work).
- **8 requirements are not implemented yet** — their tests are blocked on feature work:
  BR-025 (deployment time limit), BR-033/BR-034 (reconnect troop-control restore — the reclaim
  path is an unwired stub), BR-040–043 (deferred anyway), BR-102 (host epoch).
- **The 7 Large items:** BR-017 (needs the server simulate-from-last-state fallback built),
  BR-024 (engine AI/formation release observable only live; allied-side release needs the
  per-party ownership work), BR-025, BR-033, BR-034 (all blocked by implementation), BR-074
  (no way to force simultaneous depletion headlessly), BR-102 (blocked by implementation).
- Notable implementation-vs-requirement drift the mapping confirmed: BR-010/BR-013 (code elects
  the host at battle **entry**, not mission-ready) and BR-030 (detection semantics) — their rows
  carry `Partial` implementation for this reason.

## Execution log

**Wave 1 (2026-07-15) — the 25 `S` items: DONE.** 43 tests written TDD-style (assertions derived
from the requirement text), across 18 new test files + 4 extended, all tagged
`[Trait("Requirement", "BR-xxx")]`. Verified run: GameInterface.Tests 3/3, Coop.IntegrationTests
4/4, E2E.Tests 35 pass + 1 intended TDD-red skip. 10 tests needed test-side fixes (missing
usings, traits, baseline-relative roster assertions) — no assertion was weakened, no production
code touched. Manual-only residuals live in [BattleManualTestChecklist.md](BattleManualTestChecklist.md).

**Product finding (BR-061, TDD-red):** surrendered player-party *troops* are forfeited instead of
recorded as prisoners — `PlayerCaptivityServerHandler.cs:127-128` empties the member and prisoner
rosters after capturing the hero; the hero-as-prisoner half works. The failing test is
`BattleSurrenderTests.RecipientSurrender_RecordsSurrenderedTroops…`, skip-marked
`"BR-061 TDD red"` — unskip when fixing.

Harness lessons for later waves: never disable-patch `MapEvent.FinalizeEventAux` in an E2E
PatchScope (it already carries a production prefix — two-prefix `InvalidProgramException`, and the
broken patch state poisons every later test in the run); harness-created parties spawn with
nondeterministic rosters, so assert baseline-relative deltas, never absolute counts; fresh-worktree
restores need `/p:NuGetAudit=false` (Scriban NU1902/NU1903 audit failures in Coop.Core).

## Suggested execution order

1. **Wave 1 — the 25 `S` items:** almost all extend an existing harness pattern
   (`MissionTestEnvironment`, `HostMigrationTests`, `CoopBattleFinalizeTests`,
   `ServerBattleModeArbiterTests`); pure test-writing, no feature work.
2. **Wave 2 — the 18 `M` items:** mostly new harness capabilities (multi-client mission-flow
   fixtures, reconnection choreography, result-content assertions).
3. **Wave 3 — the 7 `L` items:** schedule alongside the implementation roadmap — each needs its
   feature (BR-017, BR-025, BR-033/034, BR-102) or live-client/manual validation (BR-024,
   BR-074) first.
